USER REPORT

All user/member dashboard pages showed:

  This section needs a moment

Related blank-page symptom on /love/login after redirect from /home:
  - URL changed to /love/login
  - React authPath state stayed null
  - App fell through to member shell (TopNav + BottomNav) with nothing in main


ERROR SOURCE
------------
Message comes from RouteErrorBoundary (src/components/RouteErrorBoundary.tsx).
Member pages are wrapped in MemberRouteBoundary in App.tsx:

  - home, discover, likes, chats, profile
  - onboarding, fast-connection, saved-profiles, voice-vibe, trusted-member

This means member pages WERE mounting, then THROWING during render — not a
network/API failure in most cases.


ROOT CAUSES IDENTIFIED
----------------------

1. BLANK PAGE ON /love/login (after redirect from /home)
   - URL changed to /love/login but React authPath state stayed null
   - App rendered member shell (TopNav + BottomNav) with empty main area
   - Trigger: visit /home unauthenticated → session restore → redirect to login
   - Direct /love/login visit worked; redirect path broke
   - Cause: redirect via navigateToPath during render before authPath synced

2. "THIS SECTION NEEDS A MOMENT" ON ALL DASHBOARD PAGES
   - RouteErrorBoundary caught render crashes on member pages
   - Pages mounted BEFORE session hydration (goToApp) finished
   - Old production code did not gate memberAccessReady on memberHydrating
   - profileComplete could be optimistically true from local cache while
     goToApp was still running

3. CONTRIBUTING RACE CONDITIONS
   - Optimistic profileComplete during blocking session restore (set before
     goToApp confirmed server state)
   - SIGNED_OUT handler kept isAuthed=true from localStorage without
     re-validating via goToApp (profileComplete could stay true)
   - TOKEN_REFRESHED could update auth while dashboards were already mounted
   - No explicit memberSessionReady flag — profileComplete alone was unreliable
   - enterMemberApp / openAppLoading not included in memberAccessReady gate

4. PRODUCTION STALE BUNDLE
   - Production served old bundle: index-BWt95Czb.js
   - Cache marker: bamsignal-v1.0.13-16-mqnyahzm
   - Fixes on main were not deployed until Coolify rebuild succeeded
   - Service worker networkFirst on /assets/ could cache old JS

5. DOCKER BUILD FAILURE (deploy blocker)
   - App.tsx referenced Signal Concierge imports never committed to main:
     * LazySignalConciergeRoutePage from lazyRoutes.ts
     * getSignalConciergeRoute from constants/signalConciergeRoutes.ts
   - tsc failed in Coolify Docker build
   - Fixed by removing incomplete concierge wiring until feature ships complete

FINAL FIX ARCHITECTURE (2ec2f0a)
--------------------------------

memberSessionReady
  - false by default
  - set true ONLY after successful goToApp() (via markMemberSessionReady)
  - cleared on logout, session restore start, enterMemberApp start

memberAccessReady (all must be true):
  - isAuthed
  - memberAppEntered
  - memberSessionReady          <-- NEW hard gate
  - !authLoading
  - !memberHydrating
  - !openAppLoading             <-- NEW
  - !isPublicSurface
  - profileComplete === true
  - !showComplianceGate

memberSurfaceBooting (shows spinner instead of pages):
  - isAuthed on member path
  - !activeAuthPath
  - authLoading OR memberHydrating OR openAppLoading OR
    !memberSessionReady OR profileComplete === null

MemberRouteBoundary
  - sessionKey={memberSessionEpoch} forces remount after session ready
  - Clears latched error boundary state from failed half-hydrated renders

SUMMARY FOR OPERATORS
---------------------
The dashboard error was NOT a random UI bug — member pages crashed because
they rendered before goToApp() finished hydrating the session. The fix adds
an explicit memberSessionReady gate so Home, Discover, Chats, and Profile
only mount after server session + profile data are confirmed. Combined with
URL-driven auth routing (be172a4) and hydration gating (e484405), this
prevents both the blank login page and the route error boundary loop.

Redeploy from main (commit 2ec2f0a or later) and verify new bundle on
production before telling users the issue is resolved.
